## What this project is

A personal MCQ quiz app for practicing CS theory (OOP, DBMS, CN, OS, ML, Agentic AI).
The user uploads `.md` files containing questions and answers. The app parses them, runs
a question-by-question quiz, saves scores, tracks wrong answers, and lets you star questions.

**This is a personal-use tool. Not public-facing. No auth. No production hardening needed.**
Keep solutions simple. Do not over-engineer.

## MD file format (the input format this app parses)

This is the format the user writes their quiz files in. The parser is built against this spec exactly.

```markdown
# Quiz Title (optional ‚Äî falls back to filename)

## Q1
Question text goes here.

A) Option one
B) Option two ‚úì
C) Option three
D) Option four

**Answer: B**

---

## Q2
Question with a code block.

```python
def example():
    return 42
```

A) It returns 42 ‚úì
B) It returns None
C) It raises an error
D) It returns "42"

**Answer: A**
```

**Parser rules:**
- Questions are delimited by `## Q{n}` headings
- Options must be on lines starting with exactly `A)`, `B)`, `C)`, `D)`
- Correct answer is on a line matching `**Answer: X**` ‚Äî this is mandatory for every question
- The `‚úì` on an option line is visual aid only ‚Äî the parser strips it and ignores it
- Code blocks use standard markdown fenced syntax with a language tag
- `---` separators between questions are optional but recommended

## Data storage rules

**All persistent data lives in `backend/data/`. Never store anything in browser localStorage.**

| File | Contents |
|---|---|
| `data/quizzes/{quiz_id}.json` | Full parsed quiz: title, question count, all questions with correct answers |
| `data/attempts/{quiz_id}.json` | List of all attempts for that quiz (score, answers, wrong_indices, timestamp) |
| `data/starred.json` | Flat array of `{ quiz_id, question_index, starred_at }` |
| `data/wrong.json` | Flat array of wrong answer records (enriched with question text, quiz title, etc.) |

**On backend startup**, `storage.ensure_data_dirs()` is called. It creates any missing directories and initialises `starred.json` and `wrong.json` with `[]` if they don't exist. The app must never crash because a data file is missing.

**The `data/` folder is in `.gitignore`.** Never commit it.

## API routes reference

### Quizzes (`routers/quizzes.py`)
```
POST   /api/quizzes/upload        Upload + parse a .md file ‚Üí saves to data/quizzes/
GET    /api/quizzes               List all quizzes (metadata only, no questions array)
GET    /api/quizzes/{quiz_id}     Get one full quiz (includes all questions + correct answers)
DELETE /api/quizzes/{quiz_id}     Delete quiz + its attempts + its stars + its wrong entries
```

### Attempts (`routers/attempts.py`)
```
POST   /api/attempts/{quiz_id}               Submit answers ‚Üí score ‚Üí save ‚Üí return result
GET    /api/attempts/{quiz_id}               All attempts for a quiz (sorted newest first)
GET    /api/attempts/{quiz_id}/{attempt_id}  One specific attempt
```

### Bookmarks (`routers/bookmarks.py`)
```
POST   /api/star        Star a question    body: { quiz_id, question_index }
DELETE /api/star        Unstar a question  body: { quiz_id, question_index }
GET    /api/starred     All starred questions (enriched with full question content)
GET    /api/wrong       All wrong answer records
DELETE /api/wrong       Clear all wrong answers
```

## Key behaviours to preserve

These are core decisions. Do not change them without a strong reason.

1. **Scoring happens on the backend** (`routers/attempts.py`), not the frontend. The backend loads the quiz's correct answers and compares them against the submitted answers. This keeps scoring authoritative and consistent.

2. **Correct answers are sent to the frontend** as part of `GET /api/quizzes/{quiz_id}`. The frontend uses them only to render the results view after submission ‚Äî not to reveal answers during the quiz. This is fine since it's personal use.

3. **Wrong answers are written to `wrong.json` at submission time** inside `POST /api/attempts/{quiz_id}`. Each wrong answer entry includes the enriched question text so the wrong answers page doesn't need to load every quiz to render.

4. **Starring is idempotent**. `POST /api/star` must check for an existing entry before inserting. Starring the same question twice must not create duplicates.

5. **Unanswered questions are scored as wrong**, not skipped. If a user submits with 3 unanswered questions, those 3 count as wrong.

6. **No deduplication on upload**. If you upload the same `.md` file twice, you get two separate quizzes with different IDs. This is intentional.

7. **No mid-quiz answer persistence**. If you refresh the Quiz page mid-quiz, your answers are lost and you start over. This is acceptable for personal use ‚Äî do not add localStorage saving for this.

## Frontend rules

### API calls
- **All API calls go through `src/api/client.js`**. No `fetch` or `axios` calls anywhere in pages or components ‚Äî only in `client.js`.
- The Vite dev proxy (`vite.config.js`) forwards `/api/*` to `http://localhost:8000`. So use `BASE = "/api"` in `client.js`, not the full URL.

### State management
- **No Redux, no Zustand, no Context API** unless a clear need arises. Use `useState` and `useEffect` local to each page.
- Data is fetched fresh on each page mount. This is fine.
- The one exception: `answers` state in `Quiz.jsx` must NOT be cleared if a submission fails. The user must be able to retry without re-answering.

### Routing
- Use `useNavigate()` for programmatic navigation after submit, upload, etc.
- Use `useParams()` to read `:quizId` and `:attemptId` from the URL.

### Error and loading states
- Every page that calls the API on mount must handle loading and error states.
- Pattern: `if (loading) return <div>Loading...</div>` / `if (error) return <div>Error: {error}</div>`
- Never silently swallow errors.

### Code blocks
- `CodeBlock.jsx` uses `react-syntax-highlighter` with `vscDarkPlus` theme and `showLineNumbers={true}`.
- Language comes from the `code_language` field on the question object (parsed from the fenced block's language tag).
- If `code_language` is null, render with `language="text"` (plain monospace, no highlighting).

## Coding conventions

### Python (backend)
- Use `pathlib.Path` for all file paths ‚Äî never string concatenation.
- Use type hints on all function signatures.
- Raise `ValueError` (not `HTTPException`) from `md_parser.py` ‚Äî the router catches it and converts to a 422 response.
- All JSON is written with `indent=2` for human readability.
- Router functions should be thin: validate input, call a service function, return the result. Business logic lives in `services/`.

### JavaScript (frontend)
- Use functional components with hooks only ‚Äî no class components.
- Use `async/await` for all API calls ‚Äî no `.then()` chains.
- Component filenames: PascalCase (`QuestionCard.jsx`). Utility files: camelCase (`client.js`).
- Keep components focused. If a component file is getting long, split it.

## What NOT to do

- **Do not add a database** (SQLite, Postgres, etc.). Flat JSON files are the deliberate choice.
- **Do not add user authentication or sessions**. This is single-user personal use.
- **Do not add Redux or a global state library** unless local state becomes genuinely unmanageable.
- **Do not use localStorage** for anything that needs to persist. All persistence is on the backend disk.
- **Do not add Docker, CI/CD, or deployment config**. This runs locally only.
- **Do not add tests** unless specifically asked. The API docs at `/docs` is the testing interface.
- **Do not paginate the API responses**. The number of quizzes and questions will always be small.
- **Do not over-style**. Simple, clean, functional. No animation libraries, no UI component libraries (no MUI, no Shadcn, etc.). Plain HTML elements with inline styles or a single CSS file is fine.

## Common tasks

### "Add a new API endpoint"
1. Add the route function to the appropriate router file (`routers/quizzes.py`, `attempts.py`, or `bookmarks.py`).
2. If it needs a new service function, add it to `services/storage.py` or `services/md_parser.py`.
3. Add the corresponding function to `frontend/src/api/client.js`.
4. Test via `http://localhost:8000/docs` before wiring up the frontend.

### "Add a new page"
1. Create the file in `frontend/src/pages/`.
2. Add the `<Route>` in `App.jsx`.
3. Add a `NavLink` in `Navbar.jsx` if it should be globally accessible.

### "The parser is failing on a quiz file"
1. Check the MD file against the format spec in this document.
2. Run `python test_parser.py` (in `backend/`) with the problematic file.
3. The error message from `ValueError` should tell you which question number is malformed and why.
4. Common issues: missing `**Answer: B**` line, options not starting with exactly `A)`, unclosed code fence.

### "Data looks wrong / needs to be reset"
- To reset everything: delete the contents of `backend/data/quizzes/`, `backend/data/attempts/`, and set `starred.json` and `wrong.json` to `[]`.
- To reset a single quiz: delete `backend/data/quizzes/{quiz_id}.json` and `backend/data/attempts/{quiz_id}.json`. Also remove entries from `starred.json` and `wrong.json` manually.
- The backend will not crash after manual data deletion ‚Äî `ensure_data_dirs()` will recreate any missing files on next startup.
